Edge computing is not just "the cloud, but closer." It represents a fundamentally different computational model — one where latency, bandwidth, and autonomy constraints reshape how we think about communication.

## Why Traditional Protocols Fall Short

Most communication protocols were designed with an implicit assumption: **reliable, high-bandwidth, low-latency networks**. At the edge, none of these assumptions hold.

Consider a fleet of autonomous vehicles sharing sensor data. You need:

- **Sub-millisecond latency** for safety-critical updates
- **Graceful degradation** when connectivity is intermittent
- **Minimal bandwidth** usage over cellular links
- **Peer-to-peer capability** when the cloud is unreachable

No single existing protocol handles all of these well.

## Key Properties for Edge Protocols

After years of working in this space, I believe effective edge protocols need:

### 1. Wire Efficiency

Every byte matters. Protocols should minimize overhead — both in message framing and in metadata. This is why Zenoh uses a compact binary wire format rather than text-based encoding.

### 2. Adaptive Routing

Data paths should adapt to network topology changes. A message between two co-located processes shouldn't traverse the cloud just because that's where the broker lives.

### 3. Computation Placement Awareness

The protocol should understand that some computations are better performed at the edge, others in the cloud. This isn't just a deployment concern — it's a protocol design concern.

## Looking Forward

As edge deployments scale from thousands to millions of nodes, the protocols we use today will need to evolve. I believe we'll see convergence toward unified protocols that handle pub/sub, querying, and storage across the full edge-to-cloud continuum.

The question isn't whether this convergence will happen — it's whether we'll design it intentionally or let it emerge chaotically from duct-taped integrations.
